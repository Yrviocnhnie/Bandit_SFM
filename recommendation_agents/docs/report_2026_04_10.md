# 2026-04-10 Phase-1 Online Learning Report

## Executive Summary

We validated a first online-learning path for the current best recommendation model:

- base model: `NeuralEncoder LinUCB`
- offline artifact: `v6_2k_neural_linear_3p3a_hardneg_other_stratified_split`
- online update scope: R/O action recommendations
- adaptation method: freeze the neural encoder and update only the per-action LinUCB heads

The phase-1 goal was not to retrain the full model online. The goal was to show that explicit user feedback can quickly personalize action ranking while keeping the representation stable.

Result:

- liked actions moved upward in rank
- disliked actions moved downward in rank
- rank-1 likes increased score margin even when rank could not improve
- a large-gap disliked action was successfully moved out of top3
- no irrelevant action appeared in top3 on the selected scenarios after adaptation

## Why This Matters

The offline model learns a strong scenario/context prior. However, real users may disagree with the offline default.

Phase 1 demonstrates a safe personalization loop:

1. show top actions for a context
2. collect a like/dislike on one action
3. map that feedback into a reward
4. update the UCB head for that action
5. propagate the update to similar contexts

Because the encoder is frozen:

- online updates are fast
- UCB statistics remain valid
- we avoid the operational risk of continuously retraining the neural representation

In production, the incoming user feedback may not include a reliable scenario id. This is compatible with the current design: we can infer the anchor scenario by searching the frozen offline-training latent space, taking the nearest contexts, and majority-voting their offline `scenario_id`. The inferred scenario is then used only to choose the local propagation region.

## Method

We used **Margin-Aware Fixed-Radius Policy v1**.

Policy settings:

| component | value |
| --- | --- |
| propagation mode | `hard-assigned-local-cutoff` |
| fixed radius | `N=2000` |
| similarity threshold | `0.80` |
| reward policy | `margin-aware-fixed-radius-v1` |
| model updated online | R/O LinUCB heads only |
| encoder | frozen |

The policy chooses feedback reward based on the score margin around the target action.

For a like:

- if the action is not already rank 1, the policy looks at the gap to the action immediately above it

For a dislike:

- if the action is not already last, the policy looks at the gap to the action immediately below it

Reward table:

| gap bin | gap range | like reward | dislike reward |
| --- | ---: | ---: | ---: |
| boundary | like rank1 or dislike last rank | `+2` | `-1` |
| small | `< 0.05` | `+2` | `-0.1` |
| medium | `0.05 - 0.5` | `+5` | `-1` |
| large | `>= 0.5` | `+10` | `-10` |

This design uses a fixed propagation radius and lets only reward strength vary with the confidence margin.

For real online data, the expected flow is:

1. encode the feedback context with the frozen encoder
2. search nearest neighbors in the offline training latent index
3. infer the anchor scenario by majority vote over neighbor `scenario_id`
4. apply the margin-aware reward policy
5. update the selected action head over the assigned local context region

This avoids requiring a separate online scenario classifier in phase 1.

## Experiment Setup

We simulated six user feedback items:

| feedback_id | scenario | feedback | target position |
| --- | --- | --- | ---: |
| `arrive_office_like_rank1` | `ARRIVE_OFFICE` | like | 1 |
| `office_lunch_out_dislike_rank1` | `OFFICE_LUNCH_OUT` | dislike | 1 |
| `office_working_like_rank2` | `OFFICE_WORKING` | like | 2 |
| `leave_office_dislike_rank2` | `LEAVE_OFFICE` | dislike | 2 |
| `cafe_stay_like_rank3` | `CAFE_STAY` | like | 3 |
| `office_to_cafe_dislike_rank3` | `OFFICE_TO_CAFE` | dislike | 3 |

The target action ids were not manually selected. For each anchor context, we first ran the frozen baseline model and then locked the action at the requested rank.

The six feedback items were applied together in one run. This is closer to the intended online setting than running each feedback independently.

## Interpretation

The experiment shows that the model can respond to user preference in the intended direction:

- small-gap likes can promote an action to rank 1
- small-gap dislikes can demote an action without fully collapsing the ranking
- rank-1 likes increase score confidence rather than changing rank
- large-gap dislikes require a strong reward and can override the offline prior

The most important product interpretation is:

> Offline labels define a good default policy. Explicit user feedback should be allowed to override that default locally.

This is why the key success metric is not only offline top3 agreement. The more relevant phase-1 metrics are:

- target action rank movement
- target action score movement
- whether replacement actions are reasonable
- same-scenario generalization
- cross-scenario drift

## Phase-2 Direction

Phase 1 used one feedback anchor per scenario. Phase 2 should test multiple feedback anchors inside the same scenario.

The target phase-2 behavior:

- in one `OFFICE_WORKING` context, the user may like action A
- in another `OFFICE_WORKING` context, the user may prefer action B
- the model should learn both preferences without forcing the entire scenario into one ranking

The current `hard-assigned-local-cutoff` mode is designed for this. It assigns same-scenario contexts to the nearest feedback anchor in frozen latent space, then applies the corresponding UCB update locally.

Recommended phase-2 evaluation:

- create 2-3 anchors within the same scenario
- use different liked/disliked target actions
- verify local rank movement near each anchor
- measure same-scenario outside-anchor drift
- measure cross-scenario drift
- add a personalized top3 metric where disliked actions are removed from the preferred set

## Recommendation

Keep the phase-1 design as the default next step:

- frozen NeuralEncoder
- online updates only to disjoint LinUCB heads
- fixed radius `N=2000`
- similarity threshold `0.80`
- margin-aware reward policy

This gives us a practical and explainable online learning path while preserving the stability of the best offline model.
