# Scenario-wise Recommendation Actions

## Notes

This file is based on the original scenario-action mapping in `default_rules.json`, and then adds extra recommendation actions where they are useful for the same scenario.

Summary:
- Total scenarios: 65 (46 existing + 19 newly added)
- Total `R&O` actions: 167 (`63` existing and `104` new; â€śnewâ€ť means not from `default_rules_hq.json`)
- Action type counts in the main table: `R` = 66, `O` = 101
- Each scenario contains at least 2 `R/O` actions in the main table, plus 1 default `A` in the separate app table (65 total)
- The `isDefault` column marks the default `R/O` in the main table, and the default app recommendation in the app table
- `A` uses the `app_category` taxonomy defined in `data_spec_for_rl.md`. There are 10 available app categories: `social`, `productivity`, `entertainment`, `navigation`, `shopping`, `news`, `health`, `music`, `reading`, `game`

Recommendation type definitions:
- `R` = Reminder. A user-facing reminder message, such as drink water, rest, prepare, or wrap up.
- `O` = Operation. A user-facing message paired with a system-side operation variable, such as `show_today_schedule` or `turn_on_dnd`.
- `A` = App recommendation. There are 10 available `app_category` values. Each `A` is a user-facing message paired with an `app_category`, indicating the intuitively most suitable default app category for the current scenario. Each scenario has 1 default `A`.

Notes:
- `R` and `O` together form the recommendation set for the scenario in the main table, covering multiple useful reminders or operations in the same context.
- In the main table, `isDefault` marks the default `R` or `O`. In the app table, it marks the default app recommendation.
- Within each scenario in the main table, actions are listed in `R -> O` order. Default app recommendations are listed separately in the app table. Within the same type, original actions come first and newly added actions come later. `trigger` is ordered as `none -> duration -> exit`.
- For `A` actions, this document recommends app categories rather than specific apps. The space of possible apps and finer-grained app-level actions is effectively unbounded, so it is not suitable to treat concrete apps as the fixed action set in this document. Instead, a separate mechanism outside the UCB agent should map an `app_category` to the most appropriate concrete app for the current user. This mapping can be learned from app usage history, or it can be explicitly configured by the user for each category, for example `music -> QQ Music`, `shopping -> Meituan`, or `navigation -> Amap`. If the current scenario recommends a `music` app, the system can use either historical usage patterns or the user's predefined preference mapping to open the most suitable music app.

Quick stats:

- Number of `R`: 66
- Number of `O`: 101
- Each scenario has 1 default `A`
